# Type Checking & Linting

## Typescript

TypeScript is configured in `tsconfig.json` for type checking only — `noEmit: true` means Vite handles transpilation and TypeScript is purely a checker.

`strict: true` enables all strict checks (`strictNullChecks`, `noImplicitAny`, `strictFunctionTypes`, etc.). On top of that:

- `noUnusedLocals` — Error on unused local variables
- `noFallthroughCasesInSwitch` — Error on switch fallthrough without explicit `break`/`return`
- `noUncheckedIndexedAccess` — Array and record bracket access returns `T \| undefined`
- `isolatedModules` — Each file must be independently transpilable (required by esbuild)

TypeScript suppression directives (`@ts-ignore` and `@ts-expect-error`) should be avoided when possible, as well as the `any` type.

## ESLint

Uses the v9 flat config format in `eslint.config.js`. The configuration is **CI-aware**: type-checked rules and stricter settings are only enabled when `CI=true`, keeping local linting fast.

ESLint is configured with a few plugins:

- `typescript-eslint` — TypeScript-aware linting
- `eslint-plugin-react` — React best practices
- `eslint-plugin-react-hooks` — Hook rules enforcement
- `eslint-plugin-react-refresh` — Vite HMR compatibility
- `@tanstack/eslint-plugin-query` — React Query best practices
- `eslint-plugin-better-tailwindcss` — Tailwind class ordering and validity

Locally, ESLint uses `tseslint.configs.recommended` (no type information needed). In CI, it switches to `tseslint.configs.recommendedTypeChecked`, unlocking rules that require the full type checker, but have a significant cost on the development environment.

Storybook stories (`src/**/*.stories.tsx`) disable `react-hooks/rules-of-hooks` since hooks are used in story decorators.

## Prettier

Configured in `.prettierrc.json`. `.prettierignore` excludes `*.generated.*` (the route tree and API types).

The `@trivago/prettier-plugin-sort-imports` plugin enforces import order across the codebase automatically on format. Groups are separated by blank lines. Specifiers within each import statement are sorted alphabetically. Side-effect-only imports are not reordered.

> **Note**: Some people like to run prettier through eslint, but things are simpler here ; these tools are both working independently.

## Spell checking

cspell is configured in `.cspell.json` and checks all `*.ts` and `*.tsx` files in `src/`. Generated files are excluded.

The configuration includes a custom dictionary for domain-specific terms (`koyeb`, `posthog`, `workos`, `seon`, etc.). Additional words can be added inline with `cspell:ignore` or `cspell:words` directives:

```ts
// cspell:ignore pgbouncer koyebdb
```

## Editor settings

`.vscode/settings.json` configures the workspace for ESLint and TypeScript:

- **ESLint flat config** mode is explicitly enabled
- **Workspace TypeScript SDK** (`node_modules/typescript/lib`) is used instead of VS Code's built-in
- **Generated files** (`*.generated.*`) are excluded from search, marked read-only, and excluded from file watching

Prettier and ESLint are pretty common, so there should be a way to integrate them with your favorite editor. For [Visual Studio Code](https://code.visualstudio.com), install the [dbaeumer.vscode-eslint](https://marketplace.visualstudio.com/items?itemName=dbaeumer.vscode-eslint) and [esbenp.prettier-vscode](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode) extensions.

## Generated files

Files matching `*.generated.*` are excluded from formatting (`.prettierignore`), spell checking (`.cspell.json`), search and editing (`.vscode/settings.json`). The two generated files are:

- `src/route-tree.generated.ts` — auto-generated by `@tanstack/router-plugin`
- `src/api/api.generated.d.ts` — API types generated from the OpenAPI spec
